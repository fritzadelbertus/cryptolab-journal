Pre-Installation: The First Obstacle

June 19, 2026

With the vision for CryptoLab taking shape, it was finally time to begin.

Before touching the operating system, I made sure to back up all of my projects to GitHub. Years of coursework, research notes, cryptographic implementations, and side projects were about to be placed at the mercy of a fresh installation. Having a backup meant I could proceed without worrying about losing anything important.

The machine chosen for the experiment was my one-year-old laptop, an Asus Vivobook 14 M1405YA. At the time, it was running Ubuntu 22.04.5 LTS, which had served me well throughout my studies and research. Soon, however, it would be replaced by a much more hands-on operating system: Arch Linux.

I visited the Arch Linux installation guide and spent some time reading through the process. Although the guide was comprehensive, I decided that the best way to learn was to dive in and work through the installation myself.

I downloaded the latest installation image, `archlinux-2026.06.01-x86_64.iso`, plugged in a blank USB flash drive, and used Rufus to create a bootable installer. Following the recommended settings, I selected the GPT partition scheme, UEFI target system, and FAT32 filesystem.

With the installation media prepared, I entered the laptop's boot configuration menu. Before proceeding, I verified several settings that would be important for the installation process. UEFI boot was enabled, Secure Boot was disabled, and Fast Boot was turned off. Everything appeared ready.

I saved the configuration and rebooted.

A few moments later, I was greeted by Arch Linux's live environment: a single terminal window and very little else.

There was no graphical installer, no welcome screen, and no obvious next button to press. Just a command prompt waiting for instructions.

The journey had officially begun.

Verifying the Environment

Following the installation guide, I first confirmed that the system had booted correctly in UEFI mode.

cat /sys/firmware/efi/fw_platform_size

The command returned 64, confirming that the system was running in 64-bit UEFI mode as expected.

The next step should have been simple: connect to the internet.

Instead, it became my first real obstacle.

The Wi-Fi Problem

On Ubuntu, I had previously relied on a USB Wi-Fi dongle because the laptop's internal wireless card never seemed to function properly. Since I was performing a fresh installation, I decided to investigate whether the issue could finally be resolved.

Unfortunately, the answer came quickly.

Running:

ip link

showed only the loopback interface, lo. No wireless device was detected.

To investigate further, I inspected the system's PCI devices:

lspci -k

Among the listed hardware was a MediaTek MT7902 Wi-Fi 6 controller.

At first, this seemed encouraging. At least the hardware was being detected.

After some research and discussion with ChatGPT, however, I discovered an unfortunate reality: the MT7902 is notoriously unsupported on Linux. While there are community efforts attempting to improve compatibility, there was no practical solution that would allow me to continue with the installation that day.

The realization was disappointing.

One of the goals of CryptoLab was to learn Linux, not to spend days reverse-engineering wireless drivers before the operating system was even installed.

After roughly thirty minutes of investigation, I made a practical decision.

The USB Wi-Fi dongle would remain.

Sometimes progress means solving a problem. Other times it means knowing which problems are not worth solving yet.

Learning to Connect from the Terminal

With the USB adapter plugged back in, networking became much easier.

Arch Linux ships with iwctl, a terminal-based wireless management tool provided by iwd. Since I had never configured Wi-Fi entirely from the command line before, this became another small learning opportunity.

The process was surprisingly straightforward:
  • Start the interface with iwctl.
  • Use device list to view available wireless devices.
  • Scan for networks using station wlan1 scan.
  • Display available networks with station wlan1 get-networks.
  • Connect using station wlan1 connect .
  • Exit the interface and verify connectivity with: ping -c 1 archlinux.org

When the ping returned successfully, I knew the system was finally online.

It was a small victory, but the first one of many.

Phase Complete

Before ending the session, I synchronized the system clock using timedatectl

The machine was now booted correctly, connected to the internet, and ready for installation.

No partitions had been touched yet. No packages had been installed. CryptoLab did not exist.

But the environment was prepared, and the first obstacle had already revealed an important lesson: building a custom Linux workstation is rarely a straight path. Even before installation begins, the hardware has its own opinions.

With the pre-installation phase complete, I was finally ready to start constructing the system itself.
